V2 main flow:

```text
Kafka / Flink
  -> Redis cache
  -> HBase durable account state
  -> ClickHouse ADS / BI
  -> Elasticsearch investigation search
  -> Great Expectations quality gate
  -> P14v2 master validation
  -> P18v2 display package
```

### 6. Key Technical Scope

| Topic | Public Scope |
| --- | --- |
| Redis | Latest-state cache, not the V2 source of truth for risk state |
| HBase | Recoverable and queryable account risk state |
| ClickHouse | V2 OLAP/BI display layer |
| Elasticsearch | V2 risk-event investigation search layer |
| Doris | V1 historical query component, excluded from V2 main validation |
| OpenSearch | Backup component, excluded from V2 main validation |
| Great Expectations | V2 main data quality gate |
| Deequ / Soda | Backup quality components, excluded from V2 main validation |
| Ranger / Atlas | Minimal governance and metadata validation boundary |
| Prometheus / Grafana | Lightweight monitoring entry and readiness reference |

### 7. Public Boundary

- The public repository keeps only display-ready documentation, source code, scripts and configuration templates.
- Local sensitive values, raw large files, runtime logs and large detail datasets must not be committed.
- V1 and V2 output directories remain isolated.
- V1 historical conclusions must not be rewritten as V2 conclusions.
- Experimental models must not be described as production AML models.
- Backup components must not be described as main-chain components.
- Long installation logs and troubleshooting records should be compressed into versions, ports, deployment locations, check commands and known boundaries.

### 8. Maintenance Rules

1. When a new phase script is added, update the script interface in `Project-Interface-Documentation_en.md` first, then update the phase overview in this file.
2. When a new V2 component is added, update the component and port tables in `Finance-Big-Data-Additional-Configuration_en.md` first, then update `Modular-Startup-Example_en.md`.
3. Display packages should copy only small Markdown, TSV, JSON and HTML materials.
4. Public documents should use repository-relative paths instead of personal machine paths.
5. Public documents must not include local sensitive configuration values, login materials or temporary debug notes.
